An input type that allows a broadcaster to push media to Smelter over [MoQ (Media over QUIC)](https://moq.dev/), a low-latency transport built on WebTransport / QUIC.

Smelter exposes a MoQ endpoint after you register the input by sending a [`register-input` request](/http-api/routes#register-input).

| Media Type | Supported Codecs |
| :--- | :--- |
| **Video** | `H.264`, `VP8`, `VP9` |
| **Audio** | `Opus`, `AAC` |

Codecs are auto-detected from the MoQ catalog announced by the broadcaster.

The MoQ connection is configured using the following parameters:
- The Smelter server address
- The configured MoQ port - defaults to `4443`, configurable via [`SMELTER_MOQ_SERVER_PORT`](/deployment/configuration#smelter_moq_server_port)
- The id the input was registered under (used as the URL path)
- The configured `authToken`, passed as a `token` query parameter

MoQ clients connect using a WebTransport URL.\
Format: `https://<smelter_ip>:<port>/<input_id>?token=<auth_token>`

<Aside>

MoQ requires TLS. To use your own certificate, set both of these environment variables:
- [`SMELTER_MOQ_TLS_CERT_FILE`](/deployment/configuration#smelter_moq_tls_cert_file) -- path to the TLS certificate file
- [`SMELTER_MOQ_TLS_KEY_FILE`](/deployment/configuration#smelter_moq_tls_key_file) -- path to the TLS private key file

If either variable is missing, the server won't start unless [`SMELTER_START_MOQ_SERVER`](/deployment/configuration#smelter_start_moq_server) is set to true.
When it is, Smelter generates a self-signed certificate (saved to `~/.smelter/moq_cert.pem` and `~/.smelter/moq_key.pem`) that is valid for 14 days.
Smelter logs the certificate's SHA-256 fingerprint at startup, and clients must be configured to trust it.
</Aside>

## Usage

To use a MoQ server input you must register it first. You can do it by sending a request like this:

<CollapsibleDetails summaryTitle='Example request' open>

```http
POST: /api/input/example/register
Content-Type: application/json

{
  "type": "moq_server",
  "auth_token": "example"
}
```
</CollapsibleDetails>

See [HTTP Routes](/http-api/routes#inputs-configuration) documentation to learn more about managing inputs.

Once the MoQ server input is registered and Smelter is listening on the configured URL, a broadcaster can push a stream to it — for the example above,
connect to `https://127.0.0.1:4443/example?token=example`.

## Reference

<CollapsibleDetails summaryTitle='Type definition' open>

```tsx
type MoqServerInput = {
  type: "moq_server";
  authToken: string;
  required?: bool;
  decoderMap?: DecoderMap;
  sideChannel?: SideChannel;
}
```
</CollapsibleDetails>

Parameters for registering a MoQ server input.

## Properties

### authToken
The token a broadcaster must pass as a `token` query parameter to connect to this input.

- **Type**: `string`

---

### required
Determines if the input stream is essential for output frame production. If set to true and the stream is delayed, Smelter will postpone output frames until the stream is received.
- **Type**: `bool`
- **Default value**: `false`

---

### decoderMap
Assigns which decoder should be used for media encoded with a specific codec. Currently, more than one decoder is supported only for `H264`.

- **Type**: [`DecoderMap`](#decodermap)

---

### sideChannel
Enable [side channel](/http-api/side-channel/overview) publishing for this input. The
external consumer reads decoded frames / audio from a Unix socket created under
`SMELTER_SIDE_CHANNEL_SOCKET_DIR`.

- **Type**: [`SideChannel`](#sidechannel)


## DecoderMap
Maps codecs to the provided decoders. This has no effect for codecs other than `H264`.

<CollapsibleDetails summaryTitle='Type definitions' open>

```tsx
type DecoderMap = {
  h264?: 'ffmpeg_h264' | 'vulkan_h264';
};
```
</CollapsibleDetails>

### Properties

#### h264
H264 decoder configuration.

- **Type**: `'ffmpeg_h264' | 'vulkan_h264'`
- **Default value**: If available `vulkan_h264` will be used, otherwise `ffmpeg_h264`
- **Supported values**:
  - `"ffmpeg_h264"` - Software decoder based on FFmpeg.
  - `"vulkan_h264"` (<Badge text="Required feature: gpu-video" variant="note" />) - Hardware-accelerated decoder. Requires GPU that supports Vulkan Video decoding.

## SideChannel
Per-track side channel configuration. See the [Side Channel overview](/http-api/side-channel/overview) for details on how decoded data is exposed and consumed.

<CollapsibleDetails summaryTitle='Type definition' open>

```tsx
type SideChannel = {
  video?: bool;
  audio?: bool;
  delayMs?: f64;
};
```
</CollapsibleDetails>

### Properties

#### video
Publish decoded RGBA video frames for this input on the side channel.

- **Type**: `bool`
- **Default value**: `false`

#### audio
Publish decoded PCM audio batches for this input on the side channel.

- **Type**: `bool`
- **Default value**: `false`

#### delayMs
Side channel delay in milliseconds. Frames are buffered for this duration ahead of when the queue consumes them, so the side-channel
subscriber receives them early and has roughly this much time to process before the frame is due.

- **Type**: `f64`
- **Default value**: `0`
